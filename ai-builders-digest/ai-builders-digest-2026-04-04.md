**LLM Knowledge Bases**

Something I'm finding very useful recently: using LLMs to build personal knowledge bases for various topics of research interest. In this way, a large fraction of my recent token throughput is going less into manipulating code, and more into manipulating knowledge (stored as markdown and images). The latest LLMs are quite good at it. So:

我最近发现一个非常实用的方法：用 LLM 为各种研究课题建立个人知识库。这样一来，我最近很大一部分 token 吞吐量不再是操作代码，而是操作知识（以 markdown 和图片的形式存储）。最新的 LLM 在这方面已经相当出色。流程如下：

**Data ingest:**
I index source documents (articles, papers, repos, datasets, images, etc.) into a raw/ directory, then I use an LLM to incrementally "compile" a wiki, which is just a collection of .md files in a directory structure. The wiki includes summaries of all the data in raw/, backlinks, and then it categorizes data into concepts, writes articles for them, and links them all. To convert web articles into .md files I like to use the Obsidian Web Clipper extension, and then I also use a hotkey to download all the related images to local so that my LLM can easily reference them.

**数据摄入：**
我将源文档（文章、论文、仓库、数据集、图片等）索引到 raw/ 目录中，然后用 LLM 逐步"编译"一个 wiki——这不过是一个目录结构中的 .md 文件集合。Wiki 包含 raw/ 中所有数据的摘要和反向链接，然后将数据分类为概念，为其撰写文章并建立链接。我喜欢用 Obsidian Web Clipper 扩展将网页文章转换为 .md 文件，同时用快捷键下载所有相关图片到本地，方便我的 LLM 轻松引用它们。

**IDE:**
I use Obsidian as the IDE "frontend" where I can view the raw data, the the compiled wiki, and the derived visualizations. Important to note that the LLM writes and maintains all of the data of the wiki, I rarely touch it directly. I've played with a few Obsidian plugins to render and view data in other ways (e.g. Marp for slides).

**IDE：**
我用 Obsidian 作为 IDE"前端"，可以查看原始数据、编译后的 wiki 和派生的可视化内容。需要注意的是，LLM 编写和维护 wiki 的所有数据，我几乎不直接接触它。我尝试过一些 Obsidian 插件来以其他方式渲染和查看数据（例如用 Marp 制作幻灯片）。

**Q&A:**
Where things get interesting is that once your wiki is big enough (e.g. mine on some recent research is ~100 articles and ~400K words), you can ask your LLM agent all kinds of complex questions against the wiki, and it will go off, research the answers, etc. I thought I had to reach for fancy RAG, but the LLM has been pretty good about auto-maintaining index files and brief summaries of all the documents and it reads all the important related data fairly easily at this ~small scale.

**问答：**
有趣的地方在于，一旦你的 wiki 足够大（例如我最近某个研究主题的 wiki 约有 100 篇文章和 40 万字），你就可以向 LLM 智能体提出各种复杂问题，它会去研究答案等等。我原本以为必须用上 fancy RAG，但 LLM 在自动维护索引文件和所有文档摘要方面表现得相当好，在这个~小规模范围内，它能够相当轻松地读取所有重要的相关数据。

**Output:**
Instead of getting answers in text/terminal, I like to have it render markdown files for me, or slide shows (Marp format), or matplotlib images, all of which I then view again in Obsidian. You can imagine many other visual output formats depending on the query. Often, I end up "filing" the outputs back into the wiki to enhance it for further queries. So my own explorations and queries always "add up" in the knowledge base.

**输出：**
我不喜欢在文本/终端中获得答案，而是喜欢让它为我渲染 markdown 文件、幻灯片（Marp 格式）或 matplotlib 图片，然后我又在 Obsidian 中查看它们。你可以想象根据查询不同会有许多其他视觉输出格式。通常，我最终会把输出结果"归档"回 wiki 中，以便进一步查询。所以我自己的探索和查询总是在知识库中"积累"。

**Linting:**
I've run some LLM "health checks" over the wiki to e.g. find inconsistent data, impute missing data (with web searchers), find interesting connections for new article candidates, etc., to incrementally clean up the wiki and enhance its overall data integrity. The LLMs are quite good at suggesting further questions to ask and look into.

**代码审查/健康检查：**
我运行过一些 LLM"健康检查"来检查 wiki，例如找出不一致的数据、（通过网络搜索）填补缺失的数据、发现有趣的联系以寻找新文章候选等，从而逐步清理 wiki 并增强其整体数据完整性。LLM 非常擅长建议进一步要问的问题和要研究的方向。

**Extra tools:**
I find myself developing additional tools to process the data, e.g. I vibe coded a small and naive search engine over the wiki, which I both use directly (in a web ui), but more often I want to hand it off to an LLM via CLI as a tool for larger queries.

**额外工具：**
我发现自己正在开发额外的工具来处理数据，例如我用 vibe coding 做了一个简单（但够用）的 wiki 搜索引擎，我可以直接使用它（在 Web UI 中），但更常见的是我想通过 CLI 将其交给 LLM 作为处理更大规模查询的工具。

**Further explorations:**
As the repo grows, the natural desire is to also think about synthetic data generation + finetuning to have your LLM "know" the data in its weights instead of just context windows.

**进一步探索：**
随着仓库的增长，很自然地会想到合成数据生成 + 微调，让你的 LLM"记住"数据到权重中，而不仅仅是通过上下文窗口。

TLDR: raw data from a given number of sources is collected, then compiled by an LLM into a .md wiki, then operated on by various CLIs by the LLM to do Q&A and to incrementally enhance the wiki, and all of it viewable in Obsidian. You rarely ever write or edit the wiki manually, it's the domain of the LLM. I think there is room here for an incredible new product instead of a hacky collection of scripts.

简而言之：从多个来源收集原始数据，然后由 LLM 编译成 .md wiki，然后由 LLM 通过各种 CLI 对其进行问答操作并逐步增强 wiki，所有内容都可以在 Obsidian 中查看。你几乎不需要手动编写或编辑 wiki，那是 LLM 的领域。我认为这里有很大的空间做一个令人难以置信的新产品，而不是一堆 hack 风格的脚本集合。

One of the biggest lessons thus far in building AI agents is you have to be brutally unsentmental in your architecture.

The rough loop of building AI agents looks something like:

构建 AI 智能体的大致循环如下：

1. Build a bunch of systems around the LLM to ensure that the agent can solve specific tasks very well

1. 在 LLM 周围构建一堆系统，确保智能体能非常好地解决特定任务

2. The model capabilities dramatically improve, rendering many of those systems redundant or even harmful

2. 模型能力大幅提升，使许多这些系统变得多余甚至有害

3. Remove prior scaffolding to get the new performance gains from the agent

3. 移除之前的脚手架，从智能体中获得新的性能提升

4. New capabilities emerge in the models that let you solve a new set of much harder problems

4. 模型中出现新能力，让你能够解决一套更困难的问题

5. Go back to step 1

5. 回到第 1 步

例如，在我们新的 Box Agent 中，从设计原始架构到最终发布，我们不得不演进智能体 harness 的多个组件，因为某些部分在模型改进时对智能体产生了不必要的约束。

The models continued to get insanely good at more complex reasoning, improvements in using search and other tools, writing code on the fly for new capabilities, improving context window performance for accuracy, and more.

模型在更复杂的推理、使用搜索和其他工具、写代码来获得新能力、提高上下文窗口性能以提升准确性等方面，继续变得极其强大。

Many of the mitigations we put in place for the Box Agent (like to appropriately find data that users were looking for, or ways of chunking text to deal with context window limitations), eventually meat we got lower quality results or meant we were overfitting for specific use-cases, as soon as the models got better.

我们为 Box Agent 设置的许多缓解措施（如适当找到用户正在查找的数据，或对文本分块以处理上下文窗口限制的方式），最终在模型变得更好后，反而导致结果质量下降或意味着我们对特定用例过度拟合。

The main lesson is always make sure you're taking advantage of the frontier capabilities and don't become nostalgic around the tech you've already built.

主要教训始终是：确保你正在利用前沿能力，不要对你已经构建的技术变得恋恋不舍。
