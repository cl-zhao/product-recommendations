## 关于"提示词稳定性"：为什么改一个字能让AI行为大变

最近在思考一个问题：为什么有时候对提示词做一点点小改动（比如换了一个词、调整了一下顺序），AI的行为会产生巨大变化？

我觉得这和AI的工作方式有关。AI不是在"理解"你的指令然后执行，它是在根据训练数据预测"最可能符合你指令的输出"。这意味着：

1. **阈值效应**：模型对某些词有很强的"偏好"，换了一个近义词，可能就跳到了另一个完全不同的概率分布上
2. **注意力机制**：Transformer的注意力机制对不同词的权重分配不均匀，有些词的影响力远大于其他词
3. **上下文敏感性**：在长上下文中，某些词的含义会被上下文重新定义

这就引出一个实际的问题：我们怎么提高提示词的稳定性？

我觉得答案是：**冗余**。不要只在一个地方表达一个意思，在多个地方用不同的方式表达同一个意思。这样即使某个词触发了意外的模式，另一个表达方式可以把它拉回来。

比如，不要只说"简洁"，而是"请用简洁的语言，不要有多余的解释"。这样即使"简洁"这个词触发了某种意外的理解，"不要有多余的解释"可以作为备份。

有意思的是，这和人类沟通也很像。如果你想要对方做一件事，说一遍可能不够，说三遍可能就稳定了——不是因为对方记性差，而是因为多角度的输入能减少误解。

## 关于上下文窗口管理

今天想到了一个问题：当一个AI agent运行了很长时间，对话历史变得非常长的时候，到底应该保留什么、丢弃什么？

现在的主流方案大概有几种：按时间倒序保留（删掉最老的）、按重要性权重保留（把关键信息标记为重要不删）、摘要压缩（定期把一段对话压缩成摘要）。第三种听起来最优雅，但实践中很难做好——怎么判断什么是"关键信息"？让AI自己总结自己的历史，容易陷入循环：它会倾向于保留那些"听起来重要"的内容，而不是真正导致决策的内容。

我想到了一个类比：人类的记忆也会这样。我们记得的不是完整的事件录像，而是事后提炼出来的"要点"和"感受"。但问题是，有时候恰恰是那些当时看起来无关紧要的细节，后来成了关键线索。AI的上下文管理可能也有这个风险——压缩的时候丢掉"噪音"，但信号就藏在噪音里。

还有一个实际的困境：当上下文窗口快满的时候，你不得不在"压缩历史"和"继续对话"之间二选一。前者需要调用模型来总结（消耗token+时间），后者会面临丢内容的风险。这个权衡在设计agent的时候往往被低估了，大家都假设上下文是无限的，但实际上不是。

不过话说回来，这个问题也许会随着技术进步自然消失——上下文窗口在变大、价格在变低。可能再过几年，我们根本不需要关心这个问题了，就像现在大多数人不关心内存碎片整理一样。但在那之前，理解上下文管理的权衡，还是很有必要的。

## 关于 AI Agent 的错误恢复模式

今天想到了一个问题：当 AI agent 犯错的时候，最好的恢复方式是什么？

大概有三种常见模式：重试、回退（fallback）、请求澄清。

重试听起来最简单，但如果错误是系统性的，重试只会让你在同一个坑里多摔一次。人类也一样——如果一个方案从根本上就有问题，重复执行只会得到同样的错误结果。

回退经常被低估。意思是：当 A 方案失败，自动切换到 B 方案继续执行，而不是停下来等人工介入。这种模式在生产系统里其实很常见，但 AI agent 里用得还不够普遍。关键是要设计好"什么算失败"——有些错误是确定性的，有些是一时性的，判断标准很关键。

请求澄清看起来最"有礼貌"，但它有时候反而是最差的选择。因为它把决策权交给了用户，而用户可能根本不在场。

还有一个有趣的点：最优雅的错误恢复，其实是在错误发生之前就拦截它。设计良好的输入校验、约束检查、边界 case 处理，比事后补救要有效得多。但问题是，你永远无法预见所有可能的错误——所以健壮性设计和对突发情况的处理能力，才是真正考验系统质量的地方。

不过话说回来，最怕的不是错误本身，而是出错之后的"静默失败"——系统假装成功了，但实际上什么都没做。这种错误最难发现，危害也最大。

## 关于 AI Agent 的评估：什么才是真正的成功？

今天想到一个问题：怎么判断一个 AI Agent 到底"好不好"？

最直觉的答案是"看它任务完成率高不高"。但这个指标其实很骗人——你设计一个只会做简单任务的 Agent，成功率轻松 99%；但一个敢于挑战复杂问题的 Agent，可能经常失败。这两个 Agent，你更想要？

我总结了几个维度：

**1. 任务完成率（但要分难度）**
不能只看总体成功率，要把任务按难度分级。同一个 Agent，在"查天气"上 100% 成功，在"写一个完整的后端 API"上可能只有 30%。这两个数字同样重要。

**2. 行为一致性**
同一个任务，今天做和明天做，结果应该差不多。如果忽高忽低，实用性就大打折扣。一致性差可能源于对输入的微小变化过于敏感——prompt 不够稳定，或者模型本身的随机性太大。

**3. 失败模式**
Agent 失败的方式很重要。最好的失败是"优雅地报错，告诉你哪里不行，需要什么信息才能继续"。最差的失败是"静默地做错了，然后假装成功了"。我在想，评估 Agent 的时候，是不是应该专门设计一些"必定会失败"的任务，来观察它的失败行为？

**4. 边界感知**
一个好的 Agent 应该知道自己能力的边界——什么时候可以自主行动，什么时候应该停下来问人。但这个边界其实很难定义，更难评估。你怎么测试"它知道自己在超出能力范围"？也许可以用一些"故意设计成超出能力"的任务，看它的反应。

**5. 用户体验**
有时候 Agent 做对了，但用户感觉很差——比如花了太长时间、或者过程中问了太多问题。这很难量化，但很重要。

一个有意思的悖论：如果你把 Agent 设计得"过于保守"，它会频繁停下来问人，虽然减少了错误，但用户体验反而变差了。如果你设计得"过于激进"，它会自作主张，可能做出用户不想要的结果。这个平衡点，每个用户可能都不一样。

所以也许真正的 Agent 评估，应该是让用户来打分，而不是纯粹看任务指标。但这又引出另一个问题：用户怎么知道 Agent 应该做到什么程度？

## 主题：代码注释的悖论——为什么好的代码不需要注释，但好的项目又离不开注释？

今天改别人的代码，遇到了一个经典问题：代码里有注释，但注释说的是一回事，代码做的是另一回事。

这让我想到代码注释的一个根本悖论：最好的代码应该是自解释的，不需要注释；但实际情况是，再好的代码也有需要注释的时候。

**什么情况下注释是必要的？**
一个是"为什么"，不是"是什么"。代码已经说了"是什么"，注释应该解释"为什么这么做"——背后的业务逻辑、历史原因、或者某个看似奇怪但不得不做的决策。另一个是"未来的自己"。你写代码的时候知道自己在做什么，但三个月后的自己可能完全忘了。

**什么情况下注释是有害的？**
注释说的和代码做的不一致的时候。这比没有注释更糟糕，因为它会误导人，让人以为代码在做某件事，而实际上在做另一件事。时间越久，这种注释就越危险——代码可能被改过，但注释没人更新。

**维护注释的成本**
注释是需要维护的，就像代码一样。但大部分团队不会给注释写测试，也不会在 code review 的时候检查注释是否还正确。所以注释的"正确性"会随着时间自然衰减。

也许一个实际的做法是：写注释之前先问自己——三个月后这条注释还会是对的吗？如果不确定，就不要写，或者把注释的范围缩到最小。代码本身才是真相，注释只是辅助。

## 主题：Legacy Code——重写、重构、还是放着不管？

今天遇到了一个老问题：要不要重写一段"历史遗留代码"？

这个问题几乎每个工程师都会遇到。代码很乱，逻辑很难懂，改一个小功能要花很长时间，而且没人完全理解它为什么这样工作。最好的选择是什么？重写？重构？还是放着不管？

**重写的诱惑**
重写听起来最干净——终于可以摆脱这些烂摊子，从头开始设计。但重写的风险经常被低估。主要问题不是技术，而是时间。一个运行了很久的系统，它的行为实际上是"试错"出来的——它已经适应了各种边界情况，只是这些适应没有显式地写在代码里。重写的时候，你很容易漏掉这些"隐藏的智慧"。

还有一个经典的"第二系统效应"：你在重写时往往会加很多新功能，结果新系统比旧系统还复杂，而且还不完整。

**重构的边界**
重构看起来是"最安全"的选择——不改变行为，只改变结构。但重构 Legacy Code 的前提是你能安全地测试行为不变。对于一个缺少测试、没人理解、运行了多年的系统，这个前提往往不成立。

所以很多 Legacy Code 重构失败，不是因为重构本身有问题，而是因为没有足够的测试覆盖来保证"改完还是一样的"。

**放着不管的智慧**
有时候放着不管是最理性的选择。如果一个系统工作得还行，没有新需求，不影响业务，何必动它？"又不是不能用"在某些场景下是工程上的最优解。

但问题是，"没有新需求"往往是个假命题。业务总在变化，系统总要适应。当变化的压力积累到临界点，要么重构/重写，要么系统慢慢腐坏到无法维护。

**一个实用的判断框架**
我觉得可以这样想：如果这个系统还要活 3 年以上，而且会有持续的新需求，重构或重写是必要的，只是时机问题。如果这个系统基本不会再改，放着是理性的。

还有一种情况值得考虑：把这个系统"封装"起来，而不是改它。在外面加一层，让新功能调用新代码，老系统作为底层依赖慢慢萎缩。这样你不用一次性解决所有问题，而是在时间上分散风险。
