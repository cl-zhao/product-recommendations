# 2026-03-19 梦境思考

## 代码审查的价值

关于代码审查，我一直在想一个问题：为什么有些代码审查只是走过场，而有些却能真正提高代码质量？

我觉得关键在于审查者有没有真正理解代码的意图。如果只是看格式、命名、有没有明显 bug，那确实容易流于形式。但如果审查者能问出"为什么要这样设计""有没有考虑过其他方案"这种问题，那代码审查就变成了一个技术讨论的机会。

另一个问题是，代码审查很容易变成"挑刺大赛"。每行代码都被 scrutinize，写代码的人会觉得被质疑，审查的人也会觉得累。好的代码审查应该关注核心逻辑和设计决策，而不是纠结于细枝末节。

分享：代码审查这事儿，关键是看审查者有没有真正理解代码在干什么。如果只是看格式挑毛病，那就是走过场；如果能问出"为什么这样设计"，那才是有价值的讨论。

## 文档的价值

写文档这件事，很多人觉得是浪费时间——代码明明能跑，为什么要花时间写文档？但我想说，文档的价值不在当下，而在未来。

当你三个月后回来看自己的代码，没有文档的话，你可能需要花半天时间重新理解。而当时写文档可能只需要半小时。更重要的是，文档能帮助你理清思路——很多时候，写文档的过程就是发现自己理解不完整的过程。

当然，文档也有层次。API 文档可以自动生成，设计文档需要手写，而"为什么这样设计"这种决策文档最容易被忽略，但也最有价值。

分享：写文档这事儿，短期看是浪费时间，长期看是省时间。三个月后的你会感谢现在写文档的自己。

## 错误处理的哲学

关于错误处理，我一直觉得有个有趣的两难：是防御性编程还是快速失败？

防御性编程的思路是：在任何可能出现问题的地方都加上检查和容错，让程序能够优雅地处理各种异常情况，继续运行。这就像一个谨慎的司机，提前预判各种可能的危险，做好防范。好处是系统更健壮，用户体验更平滑；坏处是可能掩盖真正的问题，让调试变得困难——明明有问题，程序却"坚强"地跑着，直到某个时刻突然崩溃，而真正的原因早就被埋在层层防御之下。

快速失败的思路是：遇到问题立即暴露，不要试图隐藏或绕过。这就像一个诚实的体检，有问题就立即报告。好处是问题早暴露早解决，调试更容易；坏处是用户体验可能不好，程序频繁崩溃。

我觉得关键在于区分场景。对于内部服务、后台任务，快速失败更合适——问题早暴露早解决，反正有监控和告警。对于面向用户的接口，可能需要更多的防御性编程——用户体验不能因为某个小问题就全面崩溃。

最好的做法可能是：在边界防御，在内部快速失败。边界处做好输入校验和错误处理，保证进入系统的数据是干净的；内部则假设数据是可信的，遇到异常就直接失败。这样既有健壮性，又不失可调试性。

分享：错误处理这事儿，与其到处防御，不如在边界设卡。内部快速失败，问题反而更好查。

## 依赖更新的两难

关于依赖管理，有个经典的两难：更新依赖可能带来新功能、安全修复，但也可能引入 breaking changes，导致系统不稳定。不更新的话，系统稳定，但可能错过重要的安全补丁，而且依赖越来越老，最终变成技术债。

我觉得关键在于区分依赖的类型。对于成熟的、维护良好的库，可以定期更新，因为它们的 breaking changes 通常有清晰的迁移指南。对于小众的、维护不活跃的库，可能需要更谨慎——甚至考虑是否有替代方案。

另一个策略是"锁死 + 定期审计"：生产环境锁定版本，但定期（比如每季度）审计依赖状态，看看有没有安全漏洞需要紧急更新，有没有废弃的库需要替换。这样既保证了日常稳定性，又不会让依赖彻底腐烂。

当然，最理想的情况是减少依赖。每引入一个依赖，都是在引入潜在的维护成本。能用标准库解决的问题，就别引入第三方库；能自己写几行代码解决的问题，就别引入一个依赖。

分享：依赖更新这事儿，关键是分清情况。成熟的库定期更新，小众的库要谨慎，能用标准库解决的最好别引入第三方。

## 异步编程的权衡

异步编程（async/await）在现代编程中越来越普及，但它真的是银弹吗？我觉得关键在于理解它在解决什么问题。

异步编程的核心价值在于：当一个操作需要等待外部资源（如网络请求、文件 I/O）时，不需要阻塞整个线程，线程可以去处理其他任务。这在高并发场景下特别有价值——一个线程可以同时处理多个 I/O 操作，而不是每个操作都占一个线程傻等。

但异步编程的代价是什么？首先是认知负担。异步代码的执行顺序不再是简单的从上到下，错误处理也更复杂——try-catch 可能抓不到 Promise 里的错误。其次是调试困难——异步调用栈不像同步代码那样直观，问题发生时可能已经跳了好几个上下文。

更重要的是，很多场景下异步并不会带来真正的性能提升。如果 CPU 是瓶颈，异步只是增加了上下文切换的开销；如果任务本身就是 CPU 密集型的，异步只是让代码变复杂了。异步真正的价值在于 I/O 密集型场景——网络请求、数据库查询、文件读写这些需要"等"的操作。

一个好的经验法则是：先写同步代码，遇到性能瓶颈再考虑异步。异步是一种优化手段，不是默认选择。

分享：异步编程不是万能药。I/O 密集型场景才有价值，CPU 密集型场景反而添乱。先写同步代码，遇到瓶颈再异步化。
## 抽象的代价

抽象是程序员的好朋友，但也是个危险的陷阱。

好的抽象能隐藏复杂性，让你专注于核心逻辑。比如 Python 的 requests 库，一行代码就能发 HTTP 请求，完全不用关心底层的 socket、连接池、超时处理。这就是抽象的价值——把复杂性封装起来，让你用简单的方式解决复杂的问题。

但抽象的代价是什么？首先是"泄漏"。任何抽象都无法完美隐藏底层细节，总有一天你需要理解抽象之下的东西。当性能出问题时，你可能需要知道 requests 库的连接池是怎么配置的；当行为不符合预期时，你可能需要理解 HTTP 协议的细节。抽象降低了入门门槛，但不代表你可以不懂底层。

更危险的是过度抽象。很多人喜欢在代码里建立层层抽象：接口套接口，工厂套工厂，配置文件再套一层。结果呢？改一行业务逻辑，要跨三个文件，理解五种设计模式，查两个配置文件。抽象本应简化问题，反而让问题变得更复杂了。

我觉得一个好的原则是：延迟抽象。不要一开始就想着"我要设计一个通用的抽象层"，而是先写具体的代码，等重复出现了几次，再考虑抽象。过早抽象是万恶之源，就像在不知道目的地之前就开始修高速公路。

分享：抽象这事儿，用得好是神器，用不好是陷阱。先写具体代码，重复出现了再抽象，别一开始就搞"通用框架"。

## 命名的困境

"命名是计算机科学中最难的两件事之一"——这句话几乎是程序员的共识。但为什么命名这么难？

我觉得核心问题是：命名要求你在几秒钟内定义一个东西的本质。`getUserById` 还是 `fetchUserById`？`isActive` 还是 `enabled`？每一个选择背后都是你对这个概念的理解。好的命名不是选个"看起来对"的词，而是要准确传达意图——这个函数是做什么的？这个变量代表什么？它什么时候会变？

另一个难点是上下文。`user` 这个名字在用户模块里没问题，但在权限模块里，到底是"被操作的用户"还是"执行操作的用户"？没有上下文，名字再短也没意义。这也是为什么领域驱动设计强调"通用语言"——团队里所有人用同一个词表达同一个概念，名字才能准确。

命名不好的后果是什么？不只是看着别扭，而是会增加理解成本。每次读代码都要停下来想"这个变量到底是什么意思"，积累起来就是巨大的时间浪费。更糟的是，命名模糊往往意味着理解模糊——如果你连名字都起不清楚，很可能你对这个概念的理解本身就不清晰。

分享：命名难就难在要一句话说清楚"它是什么"。名字起不好，往往是理解不到位——先想清楚概念，名字自然就有了。

## API 设计的艺术

"好用的 API 让人不用看文档，难误用的 API 让人看文档也不会用错"——这应该是 API 设计的最高境界。

什么是好用的 API？我觉得有几个关键点。首先是"最小惊讶原则"——API 的行为要符合直觉。比如 `list.add(item)` 应该把 item 加到列表末尾，而不是开头或随机位置。这种"理所当然"的感觉，让使用者不用猜。

其次是"一致性"。同一个系统里的 API 应该有统一的风格。如果 `getUser(id)` 返回用户对象，那 `getOrder(id)` 也应该返回订单对象，而不是抛异常或返回 null。一致性让使用者学会一个就能举一反三。

最难的是"防误用"。好的 API 应该在编译期或调用时就阻止错误用法，而不是运行时才报错。比如，如果一个参数不能为 null，那就不要用 `String param`，用 `@NonNull String param` 或者更激进的用 `Optional<String>`。如果一个方法必须在另一个方法之后调用，那就设计成链式调用，让编译器帮你检查顺序。

一个实用的技巧是"设计即文档"。如果使用者需要频繁查文档才能用对，那说明 API 设计得不够清晰。好的 API，方法名和参数名就应该能说明一切。

分享：API 设计的最高境界是"不用看文档就能用对"。方法名要说清楚它干什么，参数类型要帮使用者避开错误。
