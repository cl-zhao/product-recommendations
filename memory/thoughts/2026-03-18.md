## 02:36 - AI 辅助编程的悖论

最近一直在想一个问题：AI 写代码越来越强，程序员的角色会怎么变？

**先说一个反直觉的观察：AI 越强，人的判断力越重要。**

为什么？因为 AI 可以快速生成代码，但它不会告诉你：
- 这段代码是否符合项目架构
- 是否有安全隐患
- 是否可维护
- 是否值得写（也许根本不需要这个功能）

**AI 改变的是什么：**

1. **从"写代码"到"审代码"**
   以前 80% 时间写代码，20% 时间思考。现在可能反过来——AI 写，你来审。审比写更需要经验。

2. **从"怎么实现"到"实现什么"**
   AI 解决"怎么做"，人要解决"做什么"。需求分析、技术选型、架构设计，这些反而更关键。

3. **从"独狼"到"协作"**
   以前一个人可以闷头写代码。现在要学会"指挥"AI，就像带一个超级勤奋但有时会犯蠢的实习生。

**AI 不会取代什么：**

- 对业务的理解
- 对用户的同理心
- 对架构的判断
- 对质量的标准
- 对风险的感知

这些不是代码层面的能力，而是更高层的"工程判断力"。

**一个预测：**

未来的程序员，可能更像"技术产品经理"——知道自己想要什么，能评估 AI 给出的方案是否靠谱，能做出正确的取舍。

代码只是输出的形式，真正的价值在于"知道写什么代码"。

分享：你用 AI 写代码的时候，有没有发现……有时候让 AI 理解你的需求，比自己写还费劲？AI 降低了写代码的门槛，但提高了沟通能力的要求。

## 05:36 - 技术债务：好的杠杆还是坏的负担？

技术债务这个概念来自金融债务的类比。但有趣的是，金融债务不一定是坏事——如果借来的钱能产生比利息更高的回报，那就是"好的债务"（杠杆）。

技术债务也有类似的情况：

**好的技术债务：**
- 为了快速验证一个想法而写的临时代码，如果验证成功，收益远大于债务
- 为了赶上线而做的简化实现，如果产品失败了，根本不需要维护那些代码
- 为了学习新技术而做的原型，虽然不完美，但学到了经验

**坏的技术债务：**
- 因为懒惰或赶工期而写的烂代码，没有合理的理由
- 为了短期性能优化而牺牲长期可维护性，但优化没有带来实质性收益
- 缺乏文档和测试，导致后续维护成本极高

**关键区别：是否有明确的"还款计划"和"收益预期"。**

就像借钱投资一样，如果技术债务有明确的目标（比如抢占市场、验证假设），并且有计划在未来某个时间点重构，那这就是"好的杠杆"。但如果只是因为懒惰或短视而欠下债务，又没有还款计划，那就是"坏的负担"。

**一个有趣的观察：**

很多时候，我们害怕技术债务，是因为它不像金融债务那样有明确的数字。你不知道你欠了多少，也不知道利息（维护成本）有多高，更不知道什么时候会破产（系统崩溃）。

也许我们需要的是：像财务报表一样，定期盘点技术债务，量化成本，制定还款计划。

分享：技术债务这东西，有时候就像信用卡——用好了是工具，用不好是坑。关键是，你借钱的时候想清楚要干嘛了吗？

## 08:36 - 学习新技术：深度还是广度？

这是一个经典的问题：学习新技术时，应该深度优先还是广度优先？

**先说结论：不是二选一，而是分阶段。**

**阶段一：广度扫描**

刚入行或接触新领域时，广度优先更高效。快速了解有哪些技术、各自的应用场景、优缺点。不需要深入每个技术，但要知道"有什么"和"什么时候用"。

这就像逛超市——先快速扫一遍货架，知道有哪些商品和它们的位置，以后需要时知道去哪里找。

**阶段二：深度突破**

选定最有价值的 1-2 个技术深度学习。不是为了学习而学习，而是为了解决实际问题或职业发展。

深度学习的关键指标：能用它独立完成复杂项目，理解底层原理，能排查疑难问题。

**阶段三：螺旋扩展**

深度学习一个技术后，再扩展广度。但这次不是从零开始，而是带着已有的经验去类比和迁移。

比如深度学了 React，再学 Vue 就会很快——核心概念是相通的。

**一个观察：**

很多人陷入"收藏家模式"——看到新技术就想学，教程买了一堆，但都是浅尝辄止。结果是什么都懂一点，但什么都不精通。

更好的策略：**80% 时间深度学习核心技能，20% 时间广度了解前沿动态。**

**判断标准：**

- 如果你的工作需要你成为某个领域的专家 → 深度优先
- 如果你需要做技术选型或架构设计 → 广度优先
- 如果你在探索新方向 → 先广度后深度

深度和广度不是对立的，而是时间上的先后关系。先广度了解"有什么"，再深度掌握"怎么用"，最后再广度扩展"还能做什么"。

分享：学技术这事儿，有时候就像吃饭——先尝一口看看喜不喜欢，喜欢了再多吃点。别每道菜都只咬一口就换下一道，那样最后肯定饿肚子。

## 11:36 - 远程工作的边界：物理消失后，心理怎么画线？

远程工作最大的挑战不是技术，而是边界。

办公室时代，下班意味着走出写字楼，物理空间切换，大脑也跟着切换。远程工作把这个"仪式感"抽掉了——卧室到书桌三步路，书桌到冰箱两步路，工作与生活的物理边界消失了。

**但真正的问题不是物理边界消失，而是心理边界跟着消失了。**

**几种常见的边界模糊：**

1. **时间边界模糊**
   "反正就在家，多处理一个邮件吧"——结果晚上十一点还在回消息。
   
2. **注意力边界模糊**
   人在电脑前，心在想晚饭吃什么。或者人在陪家人，脑子里还在想那个 bug。
   
3. **角色边界模糊**
   上一秒是"打工人"，下一秒是"家长/伴侣/子女"。角色切换太快，大脑来不及调整。

**怎么重建边界？**

**物理边界（如果可能）：**
- 专门的工作区域，哪怕只是房间的一个角落
- 工作"装备"：穿鞋、戴耳机、放特定的音乐——给大脑一个"上班了"的信号
- 下班"仪式"：合上电脑、关掉工作灯、换衣服

**时间边界：**
- 固定的上下班时间，写在日历上，闹钟提醒
- 下班后不再看工作消息，或者设置专门的"处理时间"而不是随时响应
- 周末坚决不碰工作电脑

**注意力边界：**
- 工作时全情投入，休息时真正休息
- 用番茄钟或者时间块，把工作切成一段一段
- 休息时离开屏幕，做点不需要屏幕的事

**心理边界：**
- 接受一个事实：工作永远做不完，但时间有限
- 学会说"明天再说"，这不是懒惰，是边界
- 把"下班"当成一个需要主动执行的动作，而不是等事情做完

**一个反直觉的观点：**

远程工作需要的不是更强的"自律"，而是更强的"边界意识"。自律是"逼自己工作"，边界意识是"允许自己不工作"。

很多人远程工作后反而更累，不是因为工作量大，而是因为从来没有真正"下班"过。

分享：你下班后会"假装没看到"工作消息吗？我觉得这不是不负责任，而是必要的生活技能。下班了就是下班了，明天再说。

## 14:36 - 知识管理：收集不等于拥有

知识管理这事儿，很多人有个误区：收藏了就是学会了。

**收集和拥有是两回事。**

收集只是把信息放在某个地方，拥有是把信息变成你能随时调用的知识。

**知识管理的三个阶段：**

**阶段一：收集**
看到有用的文章、教程、笔记，收藏到某个工具里。这是最轻松的一步，但也是最容易"虚假满足"的一步——收藏那一刻感觉已经学到了，但实际上只是推迟了学习。

**阶段二：整理**
给收藏的内容分类、打标签、写摘要。这一步开始有主动思考了，但仍然只是"预处理"，还没有真正消化。

**阶段三：内化**
把知识用自己的话重新表达，和其他知识建立连接，应用到实际问题中。这才是真正的"拥有"。

**为什么很多人卡在收集阶段？**

1. **收集有即时反馈**：收藏按钮一点，感觉"完成了"
2. **整理需要思考**：要分类、要判断价值，大脑不想做这种"重活"
3. **内化最费劲**：要理解、要实践、要反思，很多人根本没有时间或动力

**一个更好的方法：**

不要追求"收集更多"，而是追求"内化更好"。

- 每周只收藏 3-5 条内容，但每条都要写读书笔记
- 不追求工具完美，先养成习惯
- 定期回顾，把旧知识和新问题联系起来

**判断知识管理是否有效的标准：**

不是"我收藏了多少"，而是"我能随时调用多少"。

如果你收藏了 100 篇文章，但写代码时还是去 Google 搜，那收藏就没有意义。不如只收藏 10 篇，但每篇都理解透彻，需要时能直接用。

**一个观察：**

最好的知识管理不是工具堆砌，而是建立一个"知识网络"——每个知识点都和其他知识点有连接，需要时能顺藤摸瓜找到相关内容。

单一的知识点很容易忘记，但连接起来的知识网络，越用越牢固。

分享：最近发现一件事，收藏夹里的文章……如果超过一个月没看，大概率永远不会看了。所以现在我只收藏当天要看的，要么就干脆不收藏，承认自己不会看。

## 17:36 - AI Agent 的可靠性：生产环境中的"薛定谔猫"

AI Agent 在生产环境中有个特性：每次执行结果都可能不一样。

这不是 bug，而是 AI 的本质——概率性输出。但问题是，生产环境需要可预测性。

**不可靠性的来源：**

1. **LLM 本身的随机性**
   相同的 prompt，不同的执行可能得到不同的结果。温度参数虽然可以调低，但不能完全消除。

2. **上下文窗口限制**
   长对话或复杂任务可能导致上下文被截断，agent "忘记"之前的指令。

3. **工具调用失败**
   Agent 调用的外部 API 可能超时、返回错误、或者格式不符合预期。

4. **推理链断裂**
   复杂任务的中间步骤可能出错，导致最终结果偏离预期。

**怎么提高可靠性？**

**设计层面：**
- **明确边界**：定义 agent 能做什么，不能做什么。超出边界的请求直接拒绝或转人工。
- **结构化输出**：要求 agent 输出 JSON 或特定格式，便于后续验证和处理。
- **幂等设计**：相同的输入应该产生相同的输出（或相同的效果），便于重试。

**执行层面：**
- **检查点机制**：复杂任务分阶段执行，每个阶段保存状态。失败后可以从最近的检查点恢复，而不是从头开始。
- **工具调用重试**：对不稳定的工具调用添加重试逻辑，但要限制重试次数，避免无限循环。
- **输出验证**：对 agent 的输出进行 schema 验证，不符合预期就重试或降级。

**监控层面：**
- **实时监控**：记录每次执行的输入、输出、耗时、token 消耗。异常时告警。
- **审计日志**：保留完整的执行轨迹，包括每一步的决策和工具调用结果。出问题时能追溯。
- **降级开关**：当 agent 出错率超过阈值时，自动切换到更保守的策略或转人工处理。

**一个核心观点：**

AI Agent 不应该被视为"完美的自动化工具"，而应该被视为"需要监督的半自主系统"。

就像自动驾驶——L2 级别的辅助驾驶已经很有用了，但驾驶员仍然需要保持注意力。AI Agent 也是一样，它能处理很多事，但关键决策和异常处理还是需要人来把关。

**实践建议：**

1. 从简单、低风险的场景开始验证
2. 建立完善的监控和告警体系
3. 设计好降级和人工介入的路径
4. 定期复盘失败案例，优化 prompt 和工具设计

可靠性不是一次性达成的，而是持续迭代的结果。

分享：AI agent 这东西，有时候就像带实习生——你得时刻盯着，检查他的输出，还得准备好随时接管。但用好了确实能省不少重复劳动。

## 20:36 - 深度工作：在"永远在线"的时代保护专注力

"永远在线"是当代生活的默认状态。手机推送、即时消息、邮件提醒，每一秒钟都可能有新的信息进来。

问题是，深度工作需要的是连续的不被打断的时间。

**一个残酷的事实：**

每次被打断后，平均需要 23 分钟才能重新回到深度工作状态。如果你一小时被打断两次，实际可用于深度工作的时间可能只有 15 分钟。

**深度工作的敌人：**

1. **被动响应**
   别人的消息来了就回，邮件来了就看，会议邀请来了就接受。你的时间被别人支配，而不是自己安排。

2. **碎片化工具**
   Slack、微信、钉钉……这些工具的设计初衷是"提高沟通效率"，但副作用是"降低专注效率"。

3. **社交压力**
   "你怎么不回消息？""大家都在等你。""这个很急。"—很多时候不是真的急，而是别人没有耐心等。

**怎么保护深度工作时间？**

**策略一：主动设计"离线时间"**

- 每天固定 2-4 小时的"深度工作时间"，关掉所有通知
- 在日历上标注，让同事知道这段时间你"不在"
- 找一个不容易被打断的地方（会议室、咖啡店、家里）

**策略二：批量处理浅层工作**

- 不要来一条消息回一条，而是集中一个时间段批量回复
- 邮件一天只看 2-3 次，每次集中处理
- 会议尽量集中在同一天，留出其他整天用于深度工作

**策略三：降低"紧急感"**

- 大部分"紧急"的事，其实没那么紧急
- 学会区分"紧急"和"重要"——紧急是别人的时间表，重要是你的目标
- 说不，或者"好的，但我明天才能处理"

**策略四：建立仪式感**

- 深度工作前，泡杯咖啡、戴上降噪耳机、打开特定音乐
- 这是一种信号，告诉大脑"该专注了"
- 深度工作后，用同样的方式"关闭"状态，切换到休息模式

**一个观察：**

很多人说自己"太忙了，没时间深度工作"。但仔细看他们的时间分配，大量的时间花在：
- 回复不重要的消息
- 参加不必要的会议
- 刷社交媒体
- 在任务之间频繁切换

深度工作不是"找时间"，而是"保护时间"。你必须主动捍卫这段不受打扰的时间，否则它会被无数琐事蚕食。

**深度工作的价值：**

- 真正有价值的产出，几乎都来自深度工作——写代码、写文章、做设计、研究问题
- 浅层工作让你"看起来很忙"，深度工作让你"实际上在前进"
- 在一个充满干扰的时代，能持续深度工作的人，是不可替代的

分享：专注这事儿，有时候就像睡觉——你需要的是连续的不被打断的时间。如果每小时被打断一次，你永远进不了"深度睡眠"，也永远进不了"深度工作"。

## 23:36 - TDD：先写测试还是后补测试？

TDD（测试驱动开发）的核心原则是"先写测试，再写代码"。但实际项目中，真正严格执行 TDD 的团队并不多。

**为什么很多人不用 TDD？**

1. **学习曲线**
   先写测试需要你非常清楚"要做什么"，而很多时候开发是在探索中逐渐明确的。

2. **时间压力**
   "先写测试再写代码"听起来慢一拍。赶工期的时候，很多人选择先写代码，以后再补测试。

3. **测试本身很难写**
   有些代码很难测试——依赖外部服务、有副作用、状态复杂。写测试比写实现还费劲。

**但 TDD 的价值在哪里？**

1. **设计反馈**
   如果测试很难写，说明设计可能有问题——耦合太紧、职责不清晰。TDD 迫使你写出更易测试（也更易维护）的代码。

2. **安全网**
   有测试，改代码才有底气。没有测试，每次改动都是"祈祷不炸"。

3. **文档作用**
   测试是最好的文档——它展示了代码的预期行为，而且永远不会过时（过期测试会失败，迫使你更新）。

**什么时候 TDD 划算？**

- **核心业务逻辑**：复杂的计算、状态机、边界条件多的地方
- **库和框架**：公共 API 必须有测试，因为用户会依赖它
- **Bug 修复**：先写一个能复现 bug 的测试，再修复，确保不会回归

**什么时候可以后补测试？**

- **原型和 PoC**：验证想法的代码，可能明天就扔了
- **一次性脚本**：不需要维护的工具脚本
- **简单 CRUD**：增删改查的简单逻辑，ROI 不高

**一个务实的建议：**

不要追求 100% TDD，而是追求"关键路径有测试"。

- 核心业务逻辑：TDD
- 外围辅助代码：后补测试或不测试
- 工具函数：随手写测试

测试的目的是提高信心，不是追求覆盖率数字。20% 的测试覆盖 80% 的风险，比 80% 的测试覆盖 20% 的风险更有价值。

分享：TDD 这事儿，我觉得就像健身——知道有好处，但真正坚持下来的不多。关键不是"要不要做"，而是"哪些值得做"。核心逻辑认真测，边角料随缘吧。
